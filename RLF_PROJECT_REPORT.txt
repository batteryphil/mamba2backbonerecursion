  - 5 prior SFT rounds already baked into this specific checkpoint

--------------------------------------------------------------------------------
  2. THE CORE PROBLEM RLF SOLVES
--------------------------------------------------------------------------------

Standard Mamba (and all SSMs) suffer from exponential hidden state decay.
The SSM update equation: h_t = A·h_{t-1} + B·x_t
The A matrix has values in (0,1), so early context decays exponentially.

Example of the failure:
  Prompt: "apples=6, price=0.50... [200 tokens of context] ...What is cost?"
  By token 200: A^200 · h_0 ≈ 0 → "apples=6" has been erased from state

This is why word problems scored 17% and logic scored 0% after 5 SFT rounds.
No amount of SFT training fixes this — it is architectural decay, not a data
problem. Training on 1 million word problems does not help if the architecture
physically cannot hold early variable values through long contexts.

The spacer tokens ("====") used in prior SFT rounds were a workaround — they
gave the SSM extra compute time but did not prevent the underlying decay. They
are now replaced by RLF.

--------------------------------------------------------------------------------
  3. RLF ARCHITECTURE
--------------------------------------------------------------------------------

Components added on top of base mamba-1.4b:

  A. LATENT MEMORY (prefix scratchpad)
     - 8 learnable tokens prepended to every input sequence
     - Initialized σ=0.02 (NOT zeros — zeros cause dead gradients)
     - These tokens are free to evolve as working memory across loops
     - Positions 0..7 = "scratch paper", never overwritten by lifeline

  B. LIFELINE RE-INJECTION
     - x_prompt = original encoding saved before loops begin
     - At every loop: prompt positions (8..) get += gate * x_prompt
     - Prevents variable decay: even at loop 6, the model still "sees"
       the original apples=6, price=0.50 at full signal strength
     - Prefix positions 0..7 untouched — free to accumulate loop state

  C. LoopRoPE (loop index geometric encoding)
     - 1D Rotary Position Embedding keyed by loop index
     - Loop 1, Loop 2, ... Loop 6 occupy geometrically distinct subspaces
     - Without this: all loops converge to identical transform (mode collapse)
     - Prevents the "loop collapse" failure seen in early 130M experiments

  D. MAMBA1 LOOP ENGINE (dedicated SSM)
     - Auxiliary Mamba SSM: d_model=2048, d_state=16, expand=1
     - 13.2M parameters, specialized for iterative state computation
     - Residual connection: x = x + mamba1_loop(x) (starting as near-identity)
     - Initialized with out_proj.weight = 0 (safe start, learns incrementally)

  E. LOOP NORM
     - RMSNorm applied after loop engine
     - Prevents bfloat16 numeric drift over 6 loop iterations

  F. LATENT BRIDGE (low-rank AE)
     - d_model → 64 → d_model (262k params, BRIDGE_RANK=64)
     - Translates loop engine activation space back to vocabulary distribution
     - bridge_up initialized to zeros → starts as identity x + 0 = x
     - Learns to map loop state to the tokens the LM head can predict

  G. HALT TOKEN (§, token 7803 in GPT-NeoX vocab)
     - Single token in existing vocabulary — no embedding resize needed
     - Appears ~0 times in code/math training data → clean blank slate
     - Model learns to predict § when reasoning chain is complete
     - MAX_LOOPS=6 hard cap prevents infinite loops

INFERENCE BEHAVIOR:
  Input: "A=42. B=A. C=B. What is C?"
  Loop 1: lifeline inject → LoopRoPE → top 24 LoRA layers → loop SSM →
           bridge → slice prefix → lm_head → predict "42"
  Loop 2: same → predict "42"
  Loop 3: same → predict "42"
  Loop 4: same → predict "§"  (HALT — done)

  Output: 3 hops correctly traced, then automatic halt.

  7b. ZERO-INIT bridge_up RECOMMENDATION

────────────────────────────────────────────────────────────────────────────────
  7c. PROPOSED ENGINE CODE (Phase 1)
────────────────────────────────────────────────────────────────────────────────

DeepThink provided an engine with three implementation bugs:

  BUG 1: forward_bottom / forward_top methods don't exist
    DeepThink's code:
      x_prompt = self.base_mamba.forward_bottom(input_ids)  ← AttributeError
      x = self.base_mamba.forward_top(x)                   ← AttributeError
    MambaLMHeadModel has no such methods.
    Our implementation: iterates self.layers[:BASE_SPLIT] and
    self.layers[BASE_SPLIT:] directly — which actually works.

  BUG 2: Missing (x, residual) tuple propagation — FATAL
    Mamba1 layers return (hidden_states, residual) tuples that must be tracked:
      correct:   x, res = layer(x, residual=res)
      DeepThink: x = self.base_mamba.forward_top(x) ← drops residual entirely
    Without residual tracking, norm_f gets wrong input → garbage logits.
    This is the exact bug our smoke test caught and fixed before training.

  BUG 3: In-place operation breaks autograd
    DeepThink's lifeline injection:
      x_out = x_current.clone()
      x_out[:, prefix_m:, :] = x_out[:, prefix_m:, :] + x_prompt  ← in-place
    In-place on gradient-dependent tensors breaks PyTorch autograd.
    Our implementation uses torch.cat (out-of-place) preserving the graph.

  ASSESSMENT: DeepThink's engine would crash immediately on forward pass.
  Our engine passed the smoke test (270M trainable, fwd+bwd clean, HALT=7803).

  STATUS: ❌ DEEPTHINK CODE WOULD NOT RUN — our implementation is correct

────────────────────────────────────────────────────────────────────────────────
  7d. -100 PADDING (IGNORE_INDEX) RECOMMENDATION
────────────────────────────────────────────────────────────────────────────────

DeepThink said:
  "If we pad targets with standard EOS tokens, the model will game the
  Cross-Entropy loss by perfectly predicting padding, ignoring reasoning.
  Must use -100 (ignore_index) to blind loss after the HALT token."

Analysis:
  Our implementation uses min(loop_i, len(chain_targets)-1) clamping.
  Short chains in batches with longer chains repeat the HALT target for
  extra loops. This IS a mild form of the loss gaming DeepThink warns about.
  However: HaltAcc=1.000 from Phase 3a step 675 — the model is not using
  this to avoid learning. Loss is dropping (13→0.33) normally.
  The -100 approach is architecturally cleaner but mid-training change
  would require restarting Phase 3b. Not worth the disruption.

  STATUS: ⚠️ VALID CONCERN, MINOR IN PRACTICE — applied to next iteration

────────────────────────────────────────────────────────────────────────────────
  7e. SEQUENCE CHAIN INTERMEDIATE STEPS
────────────────────────────────────────────────────────────────────────────────

DeepThink said:
  Generate targets [x2, x3, x4, x5, §] — all intermediate values.
  Our dataset generates [x_final, §] — only the final answer.

Analysis:
  DeepThink's approach IS better for teaching sequence induction.
  With intermediate targets, each loop must compute one step of the pattern.
  Our approach only supervises the final step, potentially allowing the model

  Worth applying in a second dataset pass if logic scores are still low
  after benchmarking.

  STATUS: ⚠️ VALID IMPROVEMENT — apply in second iteration if logic <40%

  7g. encoded[-1] vs encoded[0] TARGET TOKEN SELECTION
────────────────────────────────────────────────────────────────────────────────

DeepThink's dataset:
