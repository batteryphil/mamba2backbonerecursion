  Model:       state-spaces/mamba2-1.3b (48 layers, d_model=2048)
  Script:      finetune_mamba2_13b.py
  Key changes vs v26:
    - Frozen layers 0-23 (feature extraction, no grad)
    - LoRA (rank=8, alpha=16) on in_proj/out_proj of layers 24-47
    - Custom Mamba3ReasoningBlock (MIMO Phase Rotator, JIT TorchScript)
      - Unit-circle constraint |A|=1 for gradient stability at depth
      - Complex-valued state (real + imaginary), 2 channels, d_state=16

    - Precision: pure bfloat16 throughout (resolved NaN from fp16/fp32 mixing)

    - Fix: pure bf16 for everything (backbone, LoRA A/B, MIMO, step_emb)
    - LoRALinear.weight property: delta computed in bf16, no dtype crossing
  Training HPs:
    - BATCH_SIZE=2, ACCUM=8 (eff batch 16)
    - SEQ_LEN=256, MAX_LOOPS=6, STEPS=100,000
    - LR: 1e-3 (loop machinery), 5e-4 (LoRA)

  Result:      Stable training, Loss ~0.004 by step 1250, 99.9% in-dist accuracy

STAGE 3: DIAGNOSTIC SUITE (4-Phase Test)
--------------------------------------------------------------------------------
  Ran diagnostic_suite.py on both v26 (130m) and v27 (1.3B).

  PHASE 1 — Over-Rev Extrapolation (OOD hop lengths: 5, 7, 10 hops)
    v26:  0/3 — complete failure
    v27:  5h: 1/3, 7h: 0/5, 10h: 1/7 — near-complete failure
    Root cause: Model always runs EXACTLY 6 loops regardless of MAX_LOOPS=15.
                Cannot generalize beyond training max (N=3 hops trained, N=6 loops).

  PHASE 2 — Latent State Probe (3-hop chain X=Apple, Y=X, Z=Y)
    v27 result:
      L1 <THINK> p=1.000  top5=[<THINK>, Apple, What, Answer, Y]
      L2 <THINK> p=1.000  top5=[<THINK>, Apple, What, Answer, Y]
      L3 <THINK> p=1.000  top5=[<THINK>, Apple, Answer, What, Y]
      L4 <THINK> p=1.000  top5=[<THINK>, Apple, Answer, What, Y]
      L5 <THINK> p=1.000  top5=[<THINK>, Apple, Answer, What, X]
      L6 Apple   p=0.916
    KEY FINDING: "Apple" sits at #2 from Loop 1. The backbone already solved
    the chain. The loops are a DELAY GATE, not a computation engine.
    = The "Bouncer Effect" / Clever Hans effect confirmed.

  PHASE 3 — Dynamic Halt
    Both models: always 6 loops for 1-hop, 2-hop, AND 3-hop. Rigid rhythm.
    Root cause: <THINK> supervision trained a fixed countdown timer.

  PHASE 4 — Reality Override (counterfactual reasoning)
    Both models: 0/4. Always picks letter "A" (MMLU position bias).
    Root cause: Pointer mask over ABCD letters; "A" wins statistically.

  DIAGNOSIS: The Dense Trajectory Supervision (target=<THINK> for all
  intermediate loops) trained the model to WAIT, not to REASON.
  The architecture is mechanically correct. The training objective was wrong.


--------------------------------------------------------------------------------
STAGE 4: v28 — LATENT FORCING (current run)
--------------------------------------------------------------------------------
  Script:      finetune_mamba2_v28.py
  Core insight: Force each loop to predict an INTERMEDIATE variable, not <THINK>.

  THE FIX — Progressive Loop Supervision ("Latent Forcing"):
    For chain "A=red. B=A. C=B. What is C?":
      OLD (v27): Loop 1→<THINK>, Loop 2→<THINK>, Loop 3→<THINK>,
                 Loop 4→<THINK>, Loop 5→<THINK>, Loop 6→red
      NEW (v28): Loop 1→A  (isolate anchor variable)
                 Loop 2→B  (hop the pointer)
                 Loop 3→red (resolve and halt)
    n_loops = n_hops per sample (not fixed MAX_LOOPS).
    The Mamba block CANNOT idle — loss only decreases if it moves
    the chain pointer forward one step per tick.
    No <THINK> tokens in training at all.

  LOOP STATE ENGINE SWAP:
    v27: Custom Mamba3ReasoningBlock (MIMO Phase Rotator, JIT TorchScript)
         - Static unit-circle A/B/C geometry (|A|=1 constraint)
         - Explicit complex state (rs, is_) passed between loops
         - Slow: pure Python graph traced by TorchScript
    v28: mamba_ssm.Mamba (Mamba1 selective scan)
         - Triton-optimized CUDA kernels (~5-10x faster than JIT MIMO)
         - Data-dependent A/B/C matrices (adapts to input sequence)
         - Stateless per-loop call: x = x + mamba3_core(x)
         - out_proj zeroed at init (near-identity warmup, same as LoRA lora_B=0)
    Speed gain: 947 TPS → 1,185 TPS (+25%)

  TRAINING DATA CHANGES:
    - system2_logic_v1.json parsed at load time for chain structure
    - Multi-hop chains (3,556): progressive per-loop variable targets
    - Single-hop items (9,982): 1-target supervision from Answer: line
    - MMLU-format (10,000): free-form answer targets (no ABCD letters)
    - Counterfactuals (4): free-form reality override samples (Phase 4 fix)
    - Total: 23,542 samples, ALL targets pre-tokenized at load time

  DATA PIPELINE SPEED FIXES:
    - Pre-tokenize all chain targets at load time, not inside the training loop
    - Pre-compute ans_start positions at load time
    - Eliminates tokenizer.encode() from hot training loop
    - BATCH_SIZE 2→4, ACCUM 8→4 (same eff batch=16, less Python overhead)
    - Skip guard: if target token not in pointer mask, skip sample
      (prevents cross_entropy(logit=-inf) = inf loss)

  WARM-START:
    - LoRA weights, step_emb, loop_norm loaded from v27 checkpoint
    - mamba3_core initialized fresh (architecture changed, shapes incompatible)

  TRAINING HPs (v28):
    - BATCH_SIZE=4, ACCUM=4 (eff batch 16)
    - SEQ_LEN=256, MAX_LOOPS=6, STEPS=100,000
    - LR: 1e-3 (step_emb, loop_norm, mamba3_core, embeddings)
         5e-4 (LoRA weights)
    - AdamW, weight_decay=0.01, clip_grad_norm=1.0
    - CosineAnnealingLR, eta_min=1e-6

  METRICS (new in v28):
    - AllLoopAcc: accuracy across ALL intermediate loop targets
                  (did the model track every step of the chain?)
    - FinalAcc:   accuracy of the final loop only (is the answer correct?)
    Both matter. AllLoopAcc is the new diagnostic signal for Latent Forcing.
